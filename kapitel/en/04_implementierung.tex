\section{Three-Repository Architecture}\label{sec:repos}
The implementation is distributed across three Git repositories with clearly separated responsibilities:
\begin{itemize}
  \item \texttt{comdare-cache-engine} --- \textbf{how} measurement is done: the building-block library and the builder
  (XML config parser, \texttt{generate\_\allowbreak{}module\_\allowbreak{}from\_\allowbreak{}profile} code
  generation, permutation loop, module loader, and the seven-phase \texttt{ExperimentDriver}), the ABI
  headers, 30~SOTA profiles under \texttt{algorithm\_profiles/sota}, and the test suites; the source tree
  follows a Pitchfork layout (\texttt{apps/}, \texttt{libs/}, \texttt{libs/common}).
  \item \texttt{comdare-prt-art} --- the \textbf{probe}: the hybrid \texttt{PrtArtSearchEngine}
  (vector/map/tuple API), its ABI adapters, and its building blocks (4+1+2 allocator pools --- seven
  pools: four size-class pools A--D, one rest pool R, and two reserved value pools ---, OLC with
  reserved blocks, multi-level layout, path-oriented prefetch, density tracker, hypothesis metrics
  H1/H2/H3); the pool, prefetch, and layout components are currently descriptor skeletons without a
  real memory backend.
  \item \texttt{probst-Diplomarbeit-cache-engine} --- \textbf{what} is tested: the \texttt{messung\_driver},
  the experiment configurations, the evaluation pipeline, and the manuscript.
\end{itemize}
Orthogonally, the terminology separates two engine terms that must not be conflated. The
\texttt{CacheEngine} (\texttt{cache-engine/libs}) is the \emph{library of axis building blocks}: it
provides per axis the interchangeable design constituents as concept/CRTP components and contains \emph{no}
measurement logic. The \texttt{ExecutionEngine}, by contrast, is the execution and measurement
infrastructure lying across all axes: as the abstract measurable root (\texttt{IExecutionEngine},
Section~\ref{sec:measurement-system}) the concrete runtime adapter
\texttt{SearchAlgorithmAbiAdapter<A>} provides, via the \texttt{SearchAlgorithmAnatomy}, the OS
primitives (cache-line-aligned allocation, NUMA-conformant read pinning, coherence-aware writing); the
adapter's measurement and observer couplings are gated by
\texttt{COMDARE\_\allowbreak{}MEASUREMENT\_\allowbreak{}ON}, and the measurement aggregator is kept
host-side by the \texttt{ExperimentDriver} (a CMake option
\texttt{COMDARE\_\allowbreak{}EXPERIMENT\_\allowbreak{}MODE} exists but does not currently wire this
gating). In short: the \texttt{CacheEngine} provides the \emph{building blocks}, the
\texttt{ExecutionEngine} the \emph{tools}; this separation keeps the production path measurement-free.

\section{PRT-ART in Detail}\label{sec:prtart-impl}
Building on the known state of the art, we describe each key point of PRT-ART individually and visually.
PRT-ART contributes several key points of its own, each located at its axis (two of them on T10). On the \emph{allocator axis}, a
\emph{4+1+2 pool allocator} (four size-class pools, one rest pool, plus two reserved value pools) replaces the algorithm-hardwired
allocation with a cache-line-aligned, low-fragmentation free list. On the \emph{concurrency axis} --- itself
split into pattern (8.1) and locking mode (8.2) --- PRT-ART contributes \emph{optimistic lock coupling (OLC)
with reserved value blocks}, coordinated by a concurrency manager that handles the disciplines OLC, HTM,
STM, lock-free, RCU, and hazard pointers; reclamation is here a sub-axis of the allocator (epoch, RCU,
hazard-pointer, and QSBR strategies selectable per permutation). On the \emph{prefetch axis} a
\emph{path-oriented bundle prefetch} fetches the hot search paths ahead of time, on the \emph{layout axis} a
\emph{multi-level layout}. Particularly cache-critical is the \emph{telemetry axis}: per-node counters cause
cache-line ping-pong on multi-core systems and falsify precisely the measurements they collect. Following
the Kuehn research line~\cite{kuehn2023bplustree}, PRT-ART therefore offers cache-coherence-aware strategies
--- \emph{leaf-only counters} (the main variant), \emph{sampling} (every $N$th leaf access), and
\emph{offline recompute} (bottom-up aggregation before reordering) ---, while the classical per-node counter
remains only as a labelled anti-pattern. The density tracker and the hypothesis metrics H1/H2/H3 the probe
adds as its own telemetry organs.
These key points are \emph{building blocks of the probe repo}; of these, currently prefetch, concurrency,
and telemetry are overridden in the measurement apparatus (together with the build axis page type, the
path-compression redirect, and the ChainRef value handle), while the 4+1+2 pool allocator and the
multi-level layout run via the cache-engine default in the measurement path until their binding
(follow-up step~E6) lands (Section~\ref{sec:prtart-demo}).
Condensed into a gallery, Figure~\ref{fig:prtart-gallery} shows the six own key points of PRT-ART, each
located at its axis.

\section{Axis Implementation and Measurement-System Levels}\label{sec:axes-impl}
Each axis is implemented as a generalized interface; alongside lie the levels of the measurement system from code
generation to the seven-phase pipeline.
Each axis is a topic directory containing a two-layer \emph{concept} (a broad topic concept and a narrow
axis concept) and a CRTP base class secured by a \texttt{requires} clause. The probe-versus-default choice
is resolved for the eleven PRT-ART fallback axes at compile time via \texttt{resolve\_baustein.hpp}
(concept-checked \texttt{if constexpr} over
tag specializations), so that the hot path contains neither \texttt{virtual} calls nor runtime switches ---
the zero-cost abstraction from Section~\ref{sec:anatomy-model}. Every SOTA design and every allocator is
integrated via a thin \emph{adapter} that provides the original implementation behind the engine ABI\@:
allocator vendors are switched on or off at compile time via per-vendor include shims
(\texttt{\#if COMDARE\_AXIS\_06\_USE\_<X>} with forward stubs), SOTA building blocks via constexpr
enable flags plus \texttt{mp\_filter} (the \texttt{COMDARE\_AXIS\_*\_USE\_}/\texttt{ENABLE\_} flag
family), each with a safe fallback (e.g.\ \texttt{std::malloc} or a re-implementation), so that the
build remains independently runnable --- \texttt{COMDARE\_HAVE\_<X>} persists only as a historical
detection define; the integration of hierarchical
prefetching~\cite{zhang2025hierarchical} additionally provides a compile-time call-graph analyzer
(\texttt{p27\_bundle\_finder}, a C++23 port of the original tool \texttt{hp\_soft} that identifies
functions whose subtree footprint exceeds a threshold as bundle entry points).

The nineteen composition axes are here not abstract but each a concrete \emph{strategy concept} with several
interchangeable building blocks, from which a concrete composition picks exactly one per slot.
Figure~\ref{fig:axes-gallery} gathers all nineteen as a gallery --- per axis the short name, the sub-axis
family and a representative selection of the building blocks; the complete building-block matrix with all
origin sources is given in Appendix~\ref{app:blocks}, the SOTA contributions per axis in Table~\ref{tab:axes-overview}.
That no composition omits an organ is enforced by the \texttt{IsComposition} concept: it demands per
algorithm the assignment of exactly these nineteen organs --- the seventeen search axes plus the two
queuing axes ---, so that no composition compiles that leaves an axis open; the complete organ set is
thus not a convention but a compile-time-checked condition.

\subsection{The One Architecture in Code}\label{sec:impl-architecture}
The \emph{one} architecture from Section~\ref{sec:anatomy-model} is made visible to the outside at code level through an ABI-stable \emph{hull}. The living being
\texttt{SearchAlgorithm} and its ABI-runtime view --- the \enquote{SearchEngine} --- are merely two views
of the same entity across the module boundary, not a parallel construct: the layering
\texttt{CacheEngine}\,$\to$\,\texttt{ExecutionEngine}\,$\to$\,\texttt{SearchEngine} separates the roles (cf.\ Section~\ref{sec:repos}) ---
the \texttt{CacheEngine} as the axis building-block library, the \texttt{ExecutionEngine} as the root of
the execution and measurement tooling (OS primitives; in experiment mode the measurement aggregator over
exactly one observer interface), and the \texttt{SearchEngine} as the search-API view measured over
precisely that interface (Figure~\ref{fig:one-architecture}). Axis-less \emph{viruses} stand as siblings
beneath the same root but carry no anatomy.

\subsection{From Profile to Tier Binary: Builder and Measurement Pipeline}\label{sec:impl-pipeline}
The builder transforms each profile into a pre-compiled C++23 module; each permutation is identified by a
bit-packed \emph{permutation flag} structure that grows, with the N-phase refinement, from nine banks
(around 50~bits) to a wider \emph{permutation identifier} with fourteen banks (82~bits) and sub-bank
bit fields; the nineteen composition axes are identified in the active measurement path via the
string-serialized axis path plus fingerprint, and a constraint filter excludes invalid combinations
before each module generation. Over these building blocks runs the \emph{seven-phase measurement pipeline} (Figure~\ref{fig:pipeline}) of the
\texttt{ExperimentDriver} per series: (1)~\textbf{Enumerate} (parse XML configs, build the permutation
list), (2)~\textbf{Codegen} (one \texttt{comdare\_perm\_<fp>.cpp} per permutation), (3)~\textbf{Compile},
(4)~\textbf{Load} (LoadLibrary/dlopen of all modules), (5)~\textbf{Execute} (\texttt{create\_instance} +
profile-aware \texttt{run\_workload}), (6)~\textbf{Measure} (aggregate the binary measurement records), and
(7)~\textbf{Persist} (\texttt{measurements.csv}/\texttt{.json}); two opt-in phases hot-compile missing
modules and verify the ABI contract per module. The downstream evaluation toolchain is a sequence of small
programs --- \texttt{sample\_data\_generator} (deterministic test data without real hardware) $\to$
\texttt{messung\_driver} (drives the series) $\to$ \texttt{binary\_to\_csv} (deserializes the binary
measurement records) $\to$ \texttt{csv\_to\_latex} (booktabs tables with per-permutation-ID algorithm fact sheets)
$\to$ \texttt{diagram\_generator} (A4-aware TikZ plots) $\to$ \texttt{latex\_to\_pdf} ---; the
measurement couplings are separated from the production path by the
\texttt{COMDARE\_\allowbreak{}MEASUREMENT\_\allowbreak{}ON} gating,
so that the latter carries no measurement overhead, and continuous integration (GitLab CI plus a
mirrored GitHub workflow) runs across all three repositories.

\section{Quality Assurance: Contract Tests per Layer}\label{sec:impl-contracts}
The layer model E4--E0 with its frozen interface contracts was developed as a concept in
Section~\ref{sec:anatomy-layers}; this section describes its realization as test infrastructure. The
occasion is a concrete project experience: two implementation increments that changed several layers
at once had to be rolled back completely because their defects arose from interactions between the
levels that neither local tests nor line-by-line review made visible. For a system with a product
space of more than $10^{14}$ combinations, multi-level changes are thus practically not acceptable
for sign-off; the consequence is to complete each layer individually --- with an explicitly frozen
interface contract to the neighbouring level, its own test harness, and its own definition of done.

The carrying test principle reads: \emph{each layer is tested against a compile-time substitute of
its neighbouring level.} The contract tests are ordinary test targets of the build pipeline; the
neighbouring level is replaced by a \emph{fake} --- a template- and concept-based stand-in in the
test scope, no runtime switch in production code. A layer can therefore be finished and signed off
before the deeper level even consumes its values, and the test question of every boundary --- does
the layer pass its definition on to the next one \emph{without loss}?
(Section~\ref{sec:anatomy-layers}) --- turns from a design principle into an executable acceptance
test.

Per layer this concretizes as follows: for the definition and evaluation layer~E4 the harness checks
the translation of the declarative XML profiles into the axis-level structure --- golden fixtures
compare the axis levels emitted from a profile structurally against an expected reference, with a
fake~E3 replacing the permutation tree; neither tree construction nor build orders are needed to
sign off the E4$\to$E3 boundary. For the permutation B\textsuperscript{+} tree~E3 the harness checks
the bijection of the binary identity: from axis-level fixtures the permutation enumeration is run
and, per path, the mixed-radix-encoded binary identifier is translated back and forth (counting plus
signature round trip) --- against a fake~E2 that builds not a single binary; added to this is a
conformity gate in the full-run path that checks every imported binary against the frozen contract
before measurement. For the tier binaries~E2 the contract is the ABI itself: an emitted reference
DLL is loaded and must satisfy the frozen interface sequence --- \texttt{IAnatomyBase},
\texttt{IObservableTier}, and \texttt{IResourceControllableTier} including a correct capability
report ---, and a control request above the reported capabilities must be answered deterministically
with the frozen error value of the apply response (\texttt{apply}\,$==$\,5) instead of being
silently accepted; the accompanying fake~E1 creates and applies the runtime controls but does not
demand any consumption effect --- that belongs to the acceptance of E1. For the runtime
configuration~E1, finally, edge tests check the corner cases of the dynamic regime: that a reused
tier is fully reset between two runs, that a pool budget below the live data set is handled honestly
instead of silently violated, and that every applied control becomes visible as a named column in
the persisted result file --- checked against the CSV output, not the in-process state ---, plus the
preservation of the two-phase operation loop (discarded warm-up, measured run) as an invariant.

Completion follows the layering top-down --- E4, E3, E2, then the measurement cross-cutting
concern~M immediately before E1 ---, and the infrastructure~E0 is deliberately consolidated last
(Section~\ref{sec:anatomy-layers}); the E3$\to$E2 boundary is already golden-tested in the current
state, the remaining harnesses are fixed as the acceptance definition of the ongoing layer
completion. The cross-cutting concern~M has its own harness that gets by without any tier binary: a
synthetic measurement source is driven through the system-axis root and its values traced into the
result columns; its code shape is described in the following section.

\section{Measurement Cross-Cutting Concern: System Axes in Code}\label{sec:impl-system-axes}
The system axes --- conceptually the \emph{blood} of the apparatus, permeating all layers without
ever touching the binary identity (Section~\ref{sec:measurement-system}) --- have their own common
root in code, separate from the anatomy hierarchy: the CRTP base \texttt{SystemAxis} (file
\texttt{system\_axis.hpp}). It is built as a zero-cost abstraction --- an empty, non-polymorphic
base without virtual calls in the hot path, whose conformance is enforced by a dedicated concept ---
and translates the concept invariants into compile-time completeness gates: a \texttt{consteval}
gate checks that each of the sixteen measurement categories is assigned to exactly one of the two
regimes (nine time- and observer-based, seven counter-based PMC categories,
Section~\ref{sec:measurement-system}), and a second one rejects any system axis that mixes
categories of both regimes --- the regime-mixing ban thus breaks compilation rather than, later, the
evaluation.

The sixteen measurement categories are carried by a single \texttt{constexpr} registry
(\texttt{measurement\_\allowbreak axis\_\allowbreak registry.hpp}): per category the name and the regime derived from the
one source. A static gate enforces that the table index matches the category value, that no name is
empty, and that no regime deviates from the root assignment --- a seventeenth category or a silent
index drift thereby becomes a compilation error. The registry names are at the same time the
reporting vocabulary of the evaluation layer (the column names of the E4 reports), and a
compile-time iteration over all entries makes the registry available to metaprogramming.

The acquisition is abstracted by a vendor-neutral measurement-source interface
(\texttt{i\_\allowbreak measurement\_\allowbreak source.hpp}). \texttt{IMeasurementSource} describes a source by its
\emph{structural} capabilities --- which of the ten event semantics (cycles, instructions,
L1/L2/L3 cache, dTLB and branch misses, memory stalls, coherence invalidations, energy) it can
express at all --- and delivers per measurement interval a \texttt{MeasuredDelta} with
\emph{per-event validity}: the value zero with the validity flag set is a genuine zero measurement;
an unset flag means \emph{not measured}. The capability report never promises a measurement
guarantee --- the truth per measurement is carried by the validity flags alone. As a guaranteed,
explicitly \emph{degraded} fallback there is the wall-clock source (\texttt{WallClockSource}): it
delivers only a nanosecond proxy for the cycles and leaves all hardware events honestly invalid;
the existing PMC family is attached by an adapter that advertises only the seven structurally
expressible channels.

The honest-0 doctrine (Section~\ref{sec:measurement-system}) is programmed into these building
blocks rather than merely documented. The wall-clock system axis refuses to label its mean as a
percentile: requests for p50 through p999 are answered as invalid, since the total time divided by
the operation count is a mean, and a mean reported as a percentile would be a phantom value ---
genuine percentiles are only delivered by the HDR histogram. The cache-line utilization is collected
as a genuine utilization ratio --- used payload bytes against occupied cache lines times line
width ---, not as a raw line counter, which would be precisely the \emph{inverse} metric. Memory
footprint and fill-buffer occupancy remain invalid until a genuine peak column or a hardware source
exists (the software operation buffer of the queuing axis is no proxy for the hardware line-fill
buffer), and IPC/CPI remains invalid as long as the counter POD carries no instruction and cycle
columns.

At the evaluation end of the cross-cutting concern lies the germ of the curve fitting from
Section~\ref{sec:heuristics-curves} as a skeleton (\texttt{curve\_fit.hpp}): a measurement-axis
curve over the working-set size is fitted by least squares to the model $y = a\log_2 x + b$ --- the
form expected for cache-hierarchy curves. Here too, honest emptiness replaces a phantom fit: the fit
status distinguishes \emph{no data}, \emph{underdetermined} (fewer than two points or no spread of
the x values), and \emph{invalid data} (undefined logarithm, non-finite values) explicitly from the
success case --- never an \emph{Ok} with NaN coefficients ---, and the result reader accepts only
strictly numeric cells and counts skipped rows as a diagnostic instead of silently turning them into
phantom points.

Completed with this are the root including its gates, the registry, the source abstraction with a
degraded fallback, and the fit germ; the wiring of these cross-cutting building blocks into runner
and probe dock as well as the mapping of the registry names onto the existing columns of the result
files are the next layer increments (Section~\ref{sec:impl-contracts}). The three concept
invariants --- system axes are not permutable, do not stand in the permutation tree, and never touch
the binary identity --- are preserved by the construction itself: the root docks onto the host-side
result structures in a read-only fashion and remains invisible to the ABI boundary.

With this, the apparatus is fully described --- from the repositories through the probe and the
nineteen axes to the measurement-system levels, their layer-wise quality assurance, and the
measurement cross-cutting concern in code. Chapter~\ref{ch:eval} lays out the methodology by which
this apparatus is assessed.
